# Plan to Complete Hire Bus Feature Flow

## Overview
The hire bus feature flow is partially implemented with UI in place, but lacks data flow between pages. The navigation uses TanStack Router with search params. We need to implement dynamic data passing, form state management with TanStack Form and Zod validation, Paystack integration, local storage for hire bookings, and UI refinements.

## Current State Analysis
- **hire-fleet.tsx**: Fully implemented with vehicle selection, totals calculation, and search params for vehicles.
- **hire-checkout-page.tsx**: UI exists but hardcoded summary data; forms have no state management.
- **hire-confirmation-page.tsx**: UI exists with hardcoded booking details; needs dynamic data.
- **bookingStorage.ts**: Handles bus trip bookings; needs extension for hire bookings.
- **manage-booking.tsx**: Manages bus trip bookings; needs updates to support hire bookings.

## Required Tasks

### 1. Update Route Search Validation
- Extend `hire-checkout.tsx` route with search schema to accept booking data (vehicles, totals, trip details).
- Add similar schema to `hire-confirmation.tsx` for booking confirmation data.

### 2. Modify Navigation from Hire Fleet to Checkout
- In `hire-fleet-summary-content.tsx`, update `handleNavigate` to pass selected vehicles, totals, and search params as serialized search params.

### 3. Implement Form State in Checkout Page with TanStack Form and Zod
- Replace plain inputs with TanStack Form and Zod validation for contact details, next of kin, and special requests in `hire-checkout-page.tsx`.
- Define Zod schema for form validation (full name, email, phone, next of kin name/relationship/phone, special requests; relationship optional).
- Use shadcn UI components (e.g., Select for relationship field with options: Spouse, Parent, Sibling, Other).
- Populate the booking summary dynamically using passed data instead of hardcoded values.
- Update vehicle list, price breakdown, and trip details from search params.

### 4. Integrate Paystack Payment Gateway
- Use `usePaystack` hook from `@/lib/paystack.tsx` in `hire-checkout-page.tsx`.
- On "Pay Now" button click, initialize payment popup.
- On success, generate booking reference, save hire booking to local storage, and navigate to `/hire-confirmation` with all booking data.
- On error, show custom toast notification.

### 5. Create Hire Booking Storage
- Create new `hireBookingStorage.ts` file with `StoredHireBooking` interface (include vehicle selections, totals, dates, passenger details).
- Add functions to save/get hire bookings using separate localStorage key.
- Generate unique hire booking references (e.g., "CHT-XXXXXXXX").

### 6. Modify Manage Booking for Hire Bookings
- Create separate component `HireManageBookingPage.tsx` for hire bookings with different UI (wordings, sections, data rendering).
- Update `manage-booking-page.tsx` to dynamically render bus or hire component based on ref lookup.
- Hire component should show fleet details, charter dates, etc.

### 7. Populate Confirmation Page
- Use passed search data to replace hardcoded values in `hire-confirmation-page.tsx` (passenger name, booking reference, vehicles, dates, total paid).

### 8. Data Serialization Strategy
- Serialize complex objects (selectedVehicles, totals) as JSON strings in search params.
- Use URL-safe encoding for special characters.

### 9. UI Refinements
- Adjust `hire-checkout-page.tsx` and `hire-confirmation-page.tsx` to align with design system (consistent styling, spacing, typography; use design tokens, improve layout).
- Add loading states for payment processing.
- Improve error handling and user feedback.

### 10. Edge Cases and Validation
- Handle invalid or missing search params gracefully.
- Validate form inputs with Zod before allowing payment.
- Ensure booking reference is unique across bus and hire bookings.
- Add fallback for Paystack API key or mock mode.

## Implementation Approach
- Use Zod schemas for search param and form validation.
- Integrate TanStack Form for form management.
- Extend localStorage storage for hire bookings.
- Test flow end-to-end with sample data, including payment success/failure scenarios.
